Sumário executivo

Este documento consolida o arsenal de APIs, bancos públicos e ferramentas para construir dossiês vivos de concorrentes — dos bits de tecnologia até a tabela de taxas em PDF. Organizado em 5 camadas de coleta + 2 camadas de análise (catálogo de produtos e comparativo de preços) + 1 pipeline de síntese com LLM.

5 camadascoleta

Stack, tráfego, reviews, financeiro/ops e marketing. Cada concorrente é um vetor nessas 5 dimensões.

~40 fontesarsenal

APIs oficiais, bancos regulatórios, scrapers direcionados e agregadores de terceiros — mix de grátis e pago.

Diff temporalvalor

O sinal real está no delta: novo produto, mudança de taxa, contratação em massa, stack trocada, processo aberto.

Síntese LLMsaída

Fatos atômicos com citação → schema canônico → reducer com prompt caching → SWOT com evidência obrigatória.

Princípio norteador

Volume de fontes sem schema canônico vira ruído. Toda evidência deve ser um fato atômico tipado (fact, provider, url, snippet, timestamp). Sem isso, o LLM alucina e o dossiê apodrece em semanas.

Catálogo vivo estilo Cielo

Como mapear maquininhas, Pix, parcelamento e features de app — com changelog histórico reconstruído. A combinação de crawl + Wayback + APK teardown + ANATEL dá cobertura quase total.

Fontes por tipo de feature

Páginas oficiais de produto
  • Crawl recorrente de /produtos, /maquininhas, /pix, /taxas — HTML bruto + render
  • Wayback CDX API — histórico completo de cada URL, base para reconstruir changelog retroativo
  • Diff de PDFs de taxas/contratos — versioning + diff textual
App (features novas)
  • AppFollow / Sensor Tower / data.ai — changelog por versão e ASO
  • Play/App Store "What's new" — já parcialmente no repo, falta persistir e diffar
  • APK teardown (APKMirror + apktool/jadx) — strings e feature flags antes do lançamento público
  • Reviews por versão → sinal de "feature X apareceu na v8.2"
Maquininhas (hardware/firmware)
  • ANATEL — homologação (sch.anatel.gov.br) — catálogo oficial gratuito de todo terminal homologado, com fotos e specs
  • INPI — patentes e marcas de novos modelos
  • BACEN — arranjos de pagamento e mudanças que forçam features (Pix por aproximação, Automático, Garantido)
  • CVM/RI — apresentações trimestrais sempre listam produtos novos
Marketing & lançamento
  • Meta + Google Ads Transparency — anúncios ativos revelam features em destaque no momento
  • YouTube oficial da empresa — vídeos de lançamento, datas exatas
  • LinkedIn corporativo e de executivos — posts de product marketing
Docs técnicos & SDKs
  • developers.cielo.com.br — API reference com changelog versionado; novos endpoints = novas features
  • GitHub SDKs oficiais — CHANGELOG.md e releases
  • Postman public workspaces — coleções atualizadas
Cobertura editorial
  • Reclame Aqui, ProXXima, Mobile Time, Neofeed, Finsiders
  • Associações: ABECS, ABFintechs — calendários setoriais

Arquitetura do catálogo

  1. Entidade ProductNome, categoria, status, taxas, features[], lançada_em, fonte_primária
  2. Entidade FeatureEvent imutável{product_id, type, date, evidence[url, provider, snippet, hash], summary}
  3. Coleta diáriaPáginas oficiais, app store what's-new, ads ativos
  4. Coleta semanal/mensalWayback backfill, ANATEL, CVM/RI, APK teardown
  5. Diff engineHash do conteúdo normalizado por URL; mudança → LLM classifica em FeatureEvent tipado
  6. Timeline por produtoRender de FeatureEvent[] ordenado; filtros por tipo = "changelog histórico"
  7. Backfill inicial via Wayback3–5 anos de snapshots → histórico retroativo imediato
  8. Taxonomia de featuresNormalizer controlado: pagamento / crédito / PIX / gestão / antifraude
  9. AlertasWebhook quando surge FeatureEvent novo em produto monitorado

Schema sugerido (Supabase)

-- produtos canônicos por concorrente products ( id uuid pk, competitor_id uuid fk, slug text, -- ex: "cielo-lio-v3" name text, category text, -- maquininha | app | pix | credito | antifraude status text, -- active | deprecated | beta launched_at date, primary_url text, canonical_taxonomy text[] ) feature_events ( id uuid pk, product_id uuid fk, type text, -- launch | update | price_change | deprecation occurred_at timestamptz, summary text, evidence jsonb -- [{url, provider, snippet, hash, captured_at}] ) page_snapshots ( -- input do diff engine id uuid pk, url text, captured_at timestamptz, content_hash text, normalized_text text, raw_html_path text -- blob storage )

Análises derivadas

Price Index

MDR médio ponderado por mix de modalidade típico do segmento → score único para ranquear concorrentes.

Movimento temporal

Delta em bps semanal/mensal por modalidade. Picos = guerra de preço ou campanha agressiva.

Segmentação oculta

Comparar preço público (site) × preço de proposta comercial (mystery shopper) revela desconto real por segmento.

Condicionantes

Taxa baixa só com contrato 12m e volume mínimo — normalizar em conditions para comparar maçã com maçã.

TCO por perfil

Simular custo total para personas fixas (MEI com R$5k/mês) em todos planos → comparativo justo para o cliente.

Heatmap por bandeira

Visa × Master × Elo × Amex — identifica onde cada concorrente perde/ganha margem.

Síntese & pipeline

Volume de fontes só vira insight com schema canônico + reducer LLM + diff temporal. Esse é o coração do sistema.

  1. Schema canônico do dossiêJSON com campos fixos: product.features[], gtm.channels[], tech.stack[], ops.headcount_trend, financial.funding[], customer.pain_points[], moats[], risks[] — cada um com evidence[] citando provider + URL + trecho.
  2. Extração por providerCada provider devolve fatos atômicos tipados (fact, source, confidence, timestamp), não texto livre. Isso é o ponto de virada.
  3. Reducer com Claude + prompt cachingSchema e instruções cacheados (TTL 5min ou 1h). Só o payload de fatos muda. Reduz custo em ~90%.
  4. SWOT derivada com citação obrigatóriaSem URL/snippet → descarta. Elimina alucinação por construção.
  5. Diff temporalRoda semanal; snapshots no Supabase. O valor real está no delta.
  6. Scorecards comparativosMatriz concorrente × dimensão preenchida a partir do mesmo schema. Vira tabela/heatmap no front.
  7. Agente investigativoLoop com AI SDK v6 (stepCountIs) que olha o dossiê, identifica campos fracos/vazios e dispara providers específicos até cobertura mínima. Melhor que rodar tudo sempre.

Princípio de construção

Comece pelo diff de páginas oficiais + Wayback backfill. É grátis, dá histórico instantâneo e prova valor antes de investir em APIs pagas. Depois, adicione Similarweb e Proxycurl para trazer tráfego e headcount. Por último, o agente investigativo.
